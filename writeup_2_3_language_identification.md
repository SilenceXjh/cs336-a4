# 2.3 语言识别

## (b) 潜在问题与缓解方式

语言识别出错可能会把目标语言中的有用训练数据过滤掉，也可能把非目标语言的数据错误保留下来，从而让训练出的模型在目标语言上变弱，或者在不希望的场景中表现出混杂的多语言行为。短文本、噪声网页、混合语言文本、罗马化文本、方言和低资源语言尤其容易受到影响，因为分类器可能会对错误标签给出较高置信度。在高风险的用户产品中，我会用人工标注样本校准过滤阈值，按语言和领域分别做审计，保留各语言的评测集，监控线上失败案例，并且对低置信度或混合语言输入使用 fallback 处理，而不是直接机械丢弃。

## (c) 20 个随机 WARC 样本的人工检查

我从 `local-shared-data/CC/example.warc.gz` 中随机抽取了 20 个页面，先使用 2.2 节实现的 HTML 文本抽取函数得到纯文本，再用 fastText 的 `lid.176.bin` 模型进行语言识别。人工判断的主语言和分类器预测在这 20 个样本中都一致；样本覆盖了中文、罗马尼亚语、荷兰语、希腊语、英文、日文和俄文。20 个样本中有 7 个主要是英文，因此该样本中的英文比例是 `7/20 = 35%`。

有些英文页面的置信度只有中等甚至较低，主要原因是它们并不是连续自然语言正文，而是导航栏、产品参数、时间戳、报错信息或图片动画界面文本。例如，一个英文 GOES-R 云图动画页面被预测为英文，但分数只有 `0.2376`。基于这个样本，如果更重视英文过滤的精确率，使用 `label == "en"` 且 `score >= 0.7` 是比较合理的阈值，因为许多低置信度英文页面本身也不是高质量 prose。如果更重视召回率，可以把阈值降到 `0.6` 左右，然后依赖后续质量过滤去掉 UI 过重或正文太少的页面。
